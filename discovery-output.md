## 3. Model Routing Pattern (4 Tiers)

| Tier | Model | Use For | Rule of Thumb |
|------|-------|---------|---------------|
| 1 - Architect | **Fable 5** | System design, data models, tricky algorithms, security review, "why is this broken after 3 attempts" | Invoke deliberately, with a written spec question. Never for routine edits. |
| 2 - Senior | **Opus 4.8** | Complex multi-file refactors, integration design (Twilio, PMS APIs), agent/conversation flows, code review before launch | When the task spans 3+ files or has non-obvious edge cases. |
| 3 - Workhorse | **Sonnet 5** | 80% of daily work: features, CRUD, tests, bug fixes, API endpoints | Default model. If it fails twice on the same task, escalate one tier. |
| 4 - Sprinter | **Haiku 4.5 / Luna** | Boilerplate, renames, comments, commit messages, simple components, config files, one-liners | Anything you could do yourself in under 10 minutes. |

**Escalate up** when: same failure twice, architectural ambiguity, security-sensitive code.  
**Route down** when: the plan is written and the work is mechanical.

## 4. Token-Saving Cheat Sheet

- **Spec once, execute many**: have Tier 1 write a `PLAN.md`, then feed only the plan (not the whole conversation) to Tier 3/4.
- **Scope context with @**: reference `@file` and `@folder` explicitly instead of letting the agent trawl the repo.
- **Keep files under ~300 lines**: smaller files = smaller diffs = cheaper edits and fewer hallucinated rewrites.
- **New chat per task**: long conversations bloat every subsequent request. Finish, commit, start fresh.
- **Don't paste logs - paste the relevant 20 lines**: pre-trim stack traces and test output yourself.
- **Ask for diffs, not rewrites**: "change only the handler in X" beats "regenerate this file."
- **Batch trivia**: collect 5 small fixes into one Haiku request rather than 5 separate agent runs.
- **Write it down in `.cursorrules`**: every convention in rules is a convention you never re-explain in a prompt.

## 5. Continuous Testing Loop (max 5 cycles)

    BUILD  -> Agent implements one scoped task from PLAN.md (one feature per cycle)
    TEST   -> Run the suite + one manual happy-path check (npm test / curl / click-through)
    VERIFY -> Read the diff yourself. Does it match the plan? Any file it shouldn't have touched?
    ITERATE-> Feed the exact failure (test output, not vibes) back to the same model.

Rules:
- **Cycle budget: 5.** If it isn't working by cycle 3, escalate the model tier. If it isn't working by cycle 5, stop - the spec is wrong, not the code. Rewrite the plan with Tier 1.
- Commit after every green cycle. Never let two features share one uncommitted diff.
- Every bug fixed gets a regression test in the same cycle.

## 6. `.cursorrules` Template

    # Project rules — <project name>

    ## Stack
    - Next.js 14 (App Router), TypeScript strict, Supabase (Postgres + Auth), Stripe, Tailwind.
    - Deploy: Vercel. Emails: Resend. SMS: Twilio.

    ## Conventions
    - Server components by default; "use client" only when interactive.
    - All DB access through /lib/db - never call Supabase directly from components.
    - Zod-validate every API input at the boundary. Trust nothing from the client.
    - Money is integer cents (AUD). Dates are UTC in the DB, formatted en-AU in the UI.
    - Errors: return typed Result objects from lib functions; throw only at route level.

    ## Style
    - Small files (<300 lines). Extract when a component exceeds one responsibility.
    - No new dependencies without asking. Prefer stdlib and existing deps.
    - No comments that narrate code. Comment only non-obvious intent.

    ## Testing
    - Vitest for unit, Playwright for the critical path (signup -> pay -> core action).
    - Every bug fix ships with a regression test in the same commit.

    ## Agent behaviour
    - Make the smallest change that satisfies the task. Do not refactor unrelated code.
    - Never edit migration files that have been applied. Create new migrations.
    - Never touch .env, billing logic, or auth flows without explicitly flagging it first.
    - After each task: run typecheck + tests, report results honestly.

## 7. Cursor Pro Power-User Tips

1. **Maintain a living `PLAN.md`** - have your architect model write and update it, and point the agent at it (`@PLAN.md`) at the start of every session. This is the single highest-leverage habit for 1-2 person teams.
2. **Use Agent mode for scaffolds, inline edit (Cmd+K) for surgery** - full-agent runs on a mature codebase invite scope creep; inline edits keep changes bounded to what you're looking at.
3. **Commit before every agent run** - a clean git state means any agent misstep is one `git checkout .` away from gone, and diffs stay reviewable.
4. **Write failing tests as prompts** - paste a failing test and say "make this pass." It constrains the model far better than prose descriptions and gives you verification for free.
5. **Review diffs like a hostile senior dev** - the model optimises for plausible, not correct. Read every diff before accepting, especially auth, billing, and anything touching customer data. Ten minutes of review beats a day of production debugging.
