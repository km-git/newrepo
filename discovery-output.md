## 3. Practical model routing

Current availability and billing must be checked in Cursor before use; this report makes no pricing or benchmark claims.

| Tier | Use | Models | Rule |
|---|---|---|---|
| Architect | System design, threat models, difficult data modelling | Claude Fable 5; Claude Opus 4.8; GPT-5.6 Sol | Produce a written spec, then step down |
| Builder | Feature implementation and integration work | GPT-5.3 Codex | Work against tests and the architect spec |
| Workhorse | CRUD, tests, UI polish, small refactors | Composer 2.5; GPT-5.4; Grok 4.5 | Default for most implementation turns |
| Runtime | Product classification, extraction, summarisation | Benchmark the cheapest adequate available model | Select on a project eval set, not reputation |

Escalate one tier only after a concrete failed attempt, carrying forward the failure evidence. Do not use a frontier model for boilerplate or a cheap model for a high-stakes compliance design.

## 4. Token-saving cheat sheet

- Plan once in a short spec; execute repeated edits against it.
- Point agents at the two to five relevant files instead of loading the repository.
- Start a focused chat for each task and persist decisions in the repository.
- Filter logs and JSON to the failing lines or relevant keys.
- Require structured output and cap runtime model output.
- Keep static runtime prompts identical so provider caching can apply.
- Batch prompt evaluations against saved fixtures rather than ad-hoc live retries.

## 5. Continuous testing loop

1. **Specify:** define the user-visible success condition, data boundaries, and failure behaviour.
2. **Test first:** add unit fixtures and an integration test for the money path.
3. **Build:** implement the smallest vertical slice.
4. **Check:** run format, type, lint, unit, and sandbox integration tests.
5. **Exercise:** run the three core browser paths—signup, primary action, billing.
6. **Evaluate AI:** score at least 20 representative fixtures; no prompt/model change may regress the baseline.
7. **Preview:** manually review permissions, mobile layout, empty states, and error recovery.
8. **Observe:** deploy with error monitoring and an uptime check; review failures weekly.
9. **Iterate:** repeat for at most five focused cycles, then narrow or cut the bet.

## 6. Reusable `.cursorrules` template

The linked instructions request `.cursorrules`, but modern Cursor projects should prefer scoped project rules or `AGENTS.md`. Keep this template in each new MVP repository and adapt its commands to that project:

```text
Project: AU micro-SaaS for non-technical SMB users.
Language: AU English. Dates: DD/MM/YYYY. Currency: A$, GST treatment explicit.

Before editing:
- State the user-visible success condition.
- Read the relevant tests and integration documentation.
- Plan before schema, authentication, billing, or external API changes.

Implementation:
- Keep changes small and avoid unrelated refactors.
- Add or update tests in the same change; bug fixes start with a failing test.
- Never invent third-party API behaviour; verify docs or sandbox responses.
- Use strict typing, least-privilege access, and row-level security where applicable.

Security:
- Store no secrets in source or logs.
- Minimise personal data and redact before model calls.
- Verify webhook signatures before processing.
- NDIS products must avoid participant health data and regulated advice.

AI runtime:
- Version prompts and structured-output schemas.
- Set timeout, retry limit, max output, and safe failure behaviour.
- Require representative eval fixtures before changing prompts or models.

Completion:
- Run formatting, type checks, lint, tests, integration checks, and the core E2E path.
- Report exact commands and any untested behaviour.
```

## 7. High-impact Cursor practices

1. Separate architecture from implementation: save the reviewed plan, then build against it in a fresh task.
2. Put recurring repository-specific constraints in `AGENTS.md` or project rules, not repeated chat prompts.
3. Give agents a self-checking command that runs formatting, typing, lint, and tests together.
4. Keep one logical change per commit so agent output remains reviewable and reversible.
